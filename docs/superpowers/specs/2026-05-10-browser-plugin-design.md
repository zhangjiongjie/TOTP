# TOTP 浏览器插件设计规格

## 1. 项目目标

第一阶段优先交付一个可在 Chrome / Edge 使用的 TOTP 浏览器插件，提供接近成品的账号管理、验证码展示与同步体验。

插件需要满足以下核心目标：

- 在 `popup` 内完成主要操作，不依赖外部页面。
- 支持本地主密码解锁与加密存储。
- 支持手动添加、`otpauth://` 导入与二维码导入。
- 支持可选 WebDAV 同步；未配置时完全本地可用。
- 支持导入、导出、搜索、置顶、分组与品牌图标。
- 为后续 Web、Android、鸿蒙端复用数据格式与同步协议打基础。

## 2. 版本范围

### 2.1 第一阶段范围

第一阶段仅实现浏览器插件端，不直接实现 Web、Android、鸿蒙客户端。

交付范围包括：

- Chrome / Edge 共用一套插件代码。
- `popup` 解锁页、列表页、详情 / 编辑页、设置页。
- TOTP 生成与倒计时刷新。
- 本地主密码、加密存储、会话解锁。
- 手动录入、`otpauth://` 粘贴导入、二维码图片导入。
- 搜索、置顶、分组 / 标签、删除、编辑。
- 本地导入 / 导出。
- 可选 WebDAV 同步。

### 2.2 明确不在第一阶段的内容

- 独立 Web 页面。
- Android 客户端。
- 鸿蒙客户端。
- 浏览器输入框自动填充验证码。
- 云端账号系统。

## 3. 目标用户与主要场景

### 3.1 目标用户

- 需要在桌面浏览器中高频查看 2FA 验证码的用户。
- 已有多平台 TOTP 使用习惯，希望未来多端同步的用户。
- 对本地安全和可离线使用有要求的用户。

### 3.2 主要使用场景

- 打开插件后立即看到本地缓存的验证码列表。
- 在无网络时继续使用上次同步到本地的数据。
- 通过复制验证码快速完成登录。
- 新增账号时，优先通过二维码图片或 `otpauth://` 完成导入。
- 配置 WebDAV 后，打开插件自动拉取远端最新数据，并定时同步。
- 导出配置做备份，或导入已有配置迁移到新设备。

## 4. 信息架构

插件首版信息架构如下：

1. 解锁页
2. 主列表页
3. 添加账号入口
4. 账号详情 / 编辑页
5. 设置页
6. 同步冲突对话框
7. 导入 / 导出流程弹层

### 4.1 解锁页

职责：

- 首次使用时设置主密码。
- 已初始化时输入主密码解锁。
- 进入解锁态后加载本地密文并解密到内存。

### 4.2 主列表页

职责：

- 展示所有 TOTP 条目。
- 提供复制、搜索、分组浏览、置顶浏览。
- 展示同步状态与基础操作入口。

### 4.3 添加账号入口

支持三种路径：

- 手动添加：`issuer`、`accountName`、`secret` 等字段。
- 粘贴 `otpauth://` 文本并解析。
- 导入二维码图片并识别出 TOTP 配置。

### 4.4 详情 / 编辑页

职责：

- 查看单个条目的完整信息。
- 编辑账号字段、分组、标签、图标。
- 提供删除入口。

### 4.5 设置页

职责：

- 修改主密码。
- 配置 WebDAV。
- 手动触发导入 / 导出。
- 查看同步状态与冲突处理记录。

## 5. 交互设计

## 5.1 总体 UI 风格

首版 UI 采用混合式方案：

- 外层布局参考用户提供的移动端参考图。
- 列表条目使用更偏微软风的清爽卡片层次。
- 顶部功能图标采用偏微软风的扁平设计。
- 整体色彩以低饱和蓝灰、雾白、深海军蓝为主，不使用艳色按钮。

视觉关键词：

- 轻
- 扁平
- 克制
- 微软风
- 品牌识别清晰

### 5.2 主列表页布局

主列表页由以下区域组成：

- 顶部标题区：应用标题、账号数量。
- 顶部操作区：右上角小图标按钮，放置 `Sync`、`Settings`。
- 搜索区：搜索输入入口。
- 列表区：按分组 / 排序展示 TOTP 卡片。
- 悬浮主操作按钮：右下角悬浮 `+ Add`。

### 5.3 列表条目设计

每个条目包含：

- 品牌图标。
- `issuer`
- `accountName`
- 当前验证码。
- 右侧倒计时圆环。
- 复制反馈状态。

### 5.4 条目交互

首版采用以下交互规则：

- 点击验证码区域：直接复制当前验证码。
- 点击卡片主体其它区域：进入详情 / 编辑页。
- 右上角提供低干扰的更多菜单（`···`）。
- 更多菜单包含：`Edit`、`Move Group`、`Delete`。
- 删除操作必须二次确认。

这样设计的目的：

- 让 `Copy` 成为最短路径的高频操作。
- 保持列表视觉干净，不用常驻编辑 / 删除按钮。
- 让低频操作有稳定入口。

### 5.5 顶部功能图标

顶部功能图标包括：

- `Search`
- `Sync`
- `Settings`

图标要求：

- 偏微软风的扁平设计。
- 用简单线性或实心几何图标表达功能。
- 不使用高饱和橙色拟态按钮。
- 颜色与整体蓝灰体系一致。

### 6.2 图标匹配策略

系统采用 `issuer / host -> iconKey -> svg asset` 的映射方式。

优点：

- 手动添加时可根据 `issuer` 自动匹配品牌图标。
- `otpauth://` 导入时可自动带出图标。
- WebDAV 同步时只同步 `iconKey`，避免同步图片二进制内容。
- 未命中时可以优雅回退为默认图标。

### 6.3 回退策略

若未命中品牌图标：

- 优先使用首字母图标。
- 再回退到统一的低干扰默认图标。

## 8. 安全设计

### 8.1 主密码机制

主密码用于派生本地加密密钥。

要求：

- 插件持久化存储中只保存加密后的 `vault`。
- 解锁后密钥仅驻留内存。
- 关闭插件或会话过期后需要重新解锁。

### 8.2 本地存储安全

本地需要加密保存：

- 账号数据。
- WebDAV 凭据。
- 导出配置中的敏感信息。

### 8.3 导入导出安全

导入导出至少支持：

- 明文导出（兼容迁移场景）。
- 加密导出（推荐默认）。

导入要求：

- 先完成解锁。
- 检查数据版本与字段完整性。
- 导入后进入合并或覆盖流程。

## 9. TOTP 功能设计

### 9.1 支持协议

首版支持标准 TOTP：

- `secret`
- `issuer`
- `accountName`
- `digits`
- `period`
- `algorithm`

### 9.2 倒计时展示

倒计时通过右侧圆环与剩余时间表现。

要求：

- 列表项实时刷新。
- 即将过期时提供轻微视觉提醒。
- 避免强烈跳动或高饱和报警。

### 9.3 复制反馈

复制后需要有明确反馈：

- 按钮文案或状态变为 `Copied`
- 短时间后恢复默认状态

## 10. 导入设计

### 10.1 手动添加

必填字段：

- `issuer`
- `accountName`
- `secret`

选填字段：

- `digits`
- `period`
- `algorithm`
- `group`
- `tags`

### 10.2 `otpauth://` 导入

系统需要：

- 解析标准 `otpauth://totp/...` 格式。
- 自动填充品牌、账号、算法等字段。
- 允许用户在导入前确认与修改。

### 10.3 二维码导入

支持用户导入二维码图片。

流程：

1. 用户选择图片。
2. 插件本地解析二维码。
3. 提取 `otpauth://`。
4. 展示预览与确认界面。
5. 用户确认后保存。

## 11. 同步设计

### 11.1 同步模式

WebDAV 是可选项。

- 未配置 WebDAV：完全本地模式。
- 配置 WebDAV：本地 + 远端同步模式。

### 11.2 打开插件时的行为

每次打开 `popup` 时：

1. 先立即加载本地缓存。
2. 后台异步拉取 WebDAV 远端数据。
3. 若有新数据，则更新本地并刷新 UI。
4. 若无网络，则继续使用本地最近一次同步的数据。

### 11.3 定时同步

首版支持定时同步与手动同步：

- 定时同步：按固定间隔执行。
- 手动同步：用户点击 `Sync` 图标触发。

### 11.4 冲突策略

首版冲突策略采用保守方案：

- 如果本地和远端都在上次同步后发生变更，则判定为冲突。
- 冲突时不自动覆盖。
- 弹出对话框让用户选择：
  - 保留本地
  - 保留远端

### 11.5 离线容错

无网时：

- 所有读取操作继续基于本地缓存。
- 不阻塞验证码使用。
- 同步失败只更新状态，不影响核心功能。

## 12. 技术架构建议

首版建议采用以下技术栈：

- React
- TypeScript
- Vite
- Chrome Extension Manifest V3

### 12.1 模块划分

建议拆分为以下模块：

#### `popup-ui`

负责：

- 解锁页
- 列表页
- 详情 / 编辑页
- 设置页
- 导入导出弹层

#### `totp-core`

负责：

- TOTP 计算
- `otpauth://` 解析
- 二维码内容解析

#### `vault-core`

负责：

- 主密码派生
- 加解密
- 导入导出格式

#### `sync-core`

负责：

- WebDAV 拉取
- WebDAV 上传
- 冲突检测
- 同步状态管理

### 12.2 复用原则

后续 Web、Android、鸿蒙端尽量复用：

- 数据格式
- 导入导出格式
- 同步协议
- 品牌图标映射规则

## 13. 非功能要求

### 13.1 性能

- 打开 `popup` 后应优先秒开本地列表。
- 倒计时刷新应平滑，不造成明显卡顿。
- 搜索和分组切换在账号数较多时仍应保持流畅。

### 13.2 可靠性

- 同步失败不影响本地使用。
- 导入失败要给出清晰错误信息。
- 删除前必须确认，避免误操作。

### 13.3 可维护性

- 业务核心模块与 UI 分离。
- 品牌图标映射独立维护。
- 同步层与本地存储层解耦。

## 14. 第一阶段验收标准

当以下条件全部满足时，第一阶段设计目标可视为达成：

- 能在 Chrome / Edge 中安装并使用插件。
- 首次使用可设置主密码。
- 解锁后可看到本地账号列表。
- 可手动添加、粘贴 `otpauth://`、导入二维码图片。
- 可复制验证码，并有清晰反馈。
- 可编辑、删除、移动分组。
- 可搜索、置顶、按分组浏览。
- 可配置 WebDAV 并完成打开即拉取、定时同步、手动同步。
- 冲突时可明确选择本地或远端版本。
- 无网时仍可使用最近一次同步到本地的数据。
- 可导出与导入配置。

## 15. 后续阶段建议

第二阶段优先实现同步内核的抽离与沉淀。

建议顺序：

1. 浏览器插件端落地并稳定数据格式。
2. 抽离可复用的同步与存储核心。
3. 实现 Android 客户端。
4. 实现鸿蒙客户端。
5. 视需要补独立 Web 页面。
