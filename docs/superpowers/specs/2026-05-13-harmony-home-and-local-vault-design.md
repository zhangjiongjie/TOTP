# 鸿蒙端首页与本地数据层设计

日期：2026-05-13

## 目标

为 `TOTP` 多端工程的 HarmonyOS NEXT 客户端定义第一阶段正式交互与页面结构，范围聚焦在：

- 安全解锁优先的首页体验
- 本地数据层与本地 vault 的页面接入方式
- 与浏览器插件端一致的核心主流程
- 更贴近鸿蒙手机端习惯的视觉与操作方式

本阶段不覆盖完整 WebDAV 同步细节，也不覆盖所有高级冲突处理 UI。

## 设计原则

- 主流程与 Web 端尽量一致：
  解锁、列表、添加、编辑、删除、同步状态的用户心智保持一致。
- 交互与视觉必须更像手机应用：
  不能把浏览器 popup 直接平移到鸿蒙端。
- 安全优先：
  首页先体现解锁状态与安全状态，再进入取码列表。
- 本地优先：
  第一阶段先把本地 vault、列表和账号编辑打稳，再逐步接入远端同步。
- 鸿蒙特色适度表达：
  用沉浸式留白、柔和光感、底部浮岛操作区体现平台风格，但不过度炫技。

## 信息架构

Harmony 客户端第一阶段采用以下页面结构：

1. 解锁页
2. 首页
3. 添加账号页
4. 编辑账号页
5. 设置页

其中首页是主承载页，其他页面从首页进入。

## 首页

### 整体定位

首页采用“安全解锁优先”的结构，而不是“验证码优先”的极简列表。

用户进入应用后的第一感知应是：

- 当前保管库是否已解锁
- 当前是否处于本地模式或已配置同步
- 最近同步状态是否正常
- 然后才是账号列表与验证码操作

### 页面结构

首页从上到下分为 3 个区域：

1. 安全状态区
2. 账号列表区
3. 底部浮岛操作区

### 安全状态区

顶部不放复杂操作按钮，改成状态导向的信息头部，包含：

- 应用标题：`TOTP Authenticator`
- 解锁状态：已用主密码解锁 / 可使用生物识别快捷解锁
- 本地保管库状态：已初始化 / 空保管库 / 读取异常
- 最近同步状态：未配置同步 / 最近同步成功 / 同步失败

状态区的重点不是展示所有配置细节，而是快速建立安全感与当前状态感。

### 账号列表区

账号列表延续 Web 端的大卡片信息结构，但适配手机端纵向浏览。

每个列表项包含：

- 品牌图标
- `issuer`
- `accountName`
- `group`
- 当前验证码
- 倒计时环
- 编辑入口

交互规则：

- 点击验证码区域：复制验证码
- 点击编辑图标：进入编辑页
- 不在首页保留 `Move Group / Delete` 菜单

列表为空时展示空状态卡片，文案强调“本地保管库已就绪，下一步可添加账号”。

## 解锁流程

### 入口策略

鸿蒙端采用“主密码为主，生物识别为快捷入口”的策略。

### 首次进入

首次进入流程为：

1. 创建主密码
2. 可选启用生物识别
3. 可选配置 WebDAV
4. 进入首页

### 日常进入

日常进入时：

- 优先展示生物识别快捷入口
- 同时保留主密码输入入口
- 生物识别失败时，不打断流程，回退到密码输入

这样既保留手机端便捷性，也不丢失跨端一致的主密码模型。

## 底部浮岛操作区

### 设计方向

底部采用常驻“胶囊浮岛”操作区，不把 `添加 / 同步 / 设置` 放在顶部栏。

这是鸿蒙端区别于 Web popup 的关键视觉点。

### 结构

浮岛常驻 3 个入口：

- 添加
- 同步
- 设置

### 视觉语言

浮岛采用：

- 半透明胶囊底
- 柔和高光
- 轻微景深
- 圆润边角
- 低饱和蓝灰体系

不使用艳色，不做霓虹质感，不做过度动画。

### 行为

- 点击 `添加`：进入添加账号页
- 点击 `同步`：触发手动同步，并在首页状态区反馈结果
- 点击 `设置`：进入设置页

## 添加账号页

添加页延续已经在 Web 端确认过的统一表单方案，但用手机端布局重做。

### 页面结构

1. 顶部标题与返回
2. 二维码/图片/链接输入入口
3. 主表单
4. 底部固定保存按钮

### 数据来源整合

添加页不拆成多个模式页，而是统一一个页面承接多种输入来源：

- 自动识别二维码
- 上传二维码图片
- 手动填写 `otpauth://`
- 手动填写表单字段

### 表单规则

必填：

- `Issuer`
- `Account`
- `Secret`

默认值字段：

- `Digits = 6`
- `Period = 30`
- `Algorithm = SHA1`
- `Group = Default`

扫码、图片解析或 `otpauth://` 若带有这些值，则自动覆盖默认值。

### 保存规则

无论数据来源是什么，识别成功后都只回填表单，不直接入库。

只有用户点击 `保存` 后，才真正写入本地 vault。

## 编辑账号页

编辑页与添加页保持相同的表单结构和手机端节奏。

### 页面结构

- 顶部标题与返回
- 可滚动表单区域
- 底部固定操作栏

### 底部操作栏

包含两个主按钮：

- `删除`
- `保存`

删除需要二次确认。

保存成功或删除确认成功后，统一返回首页。

### 字段策略

`Group` 作为普通字段放在编辑表单内，不再单独提供 `Move Group` 入口。

## 设置页

设置页第一阶段先承接与首页强相关的能力：

- WebDAV 配置入口
- 是否启用生物识别
- 是否保持浏览器重启前的解锁等跨端策略映射
- 导入导出入口
- 同步状态与最近错误

设置页不是本阶段主要视觉焦点，但信息组织应与首页状态区保持一致。

## 本地数据层

### 第一阶段存储方案

Harmony 端第一阶段使用 `Preferences` 持久化本地 vault。

原因：

- 与当前需求规模匹配
- 接入简单
- 适合本地小体量持久化
- 能作为后续加密 vault 接入前的稳定落脚点

### 存储方式

第一阶段将整个本地 vault 作为单个 JSON 快照写入本地：

- 单一存储 key
- 包含 `schemaVersion`
- 包含 `updatedAt`
- 包含 `accounts[]`

### 与后续演进的关系

这一层只作为 Harmony 端本地 façade，不应直接绑死在页面里。

后续演进方向：

- 接入共享 `packages/totp-core`
- 接入加密 vault
- 接入 `packages/totp-sync`

## 与 Web 端的一致性

需要保持一致的部分：

- 主密码心智
- 账号字段定义
- 添加/编辑表单规则
- 二维码识别后先确认再保存
- 删除需确认
- 同步状态要明确反馈

允许不同的部分：

- 顶部不保留 Web 端的 `Sync / Settings` 小图标
- 鸿蒙端使用底部浮岛替代顶部操作栏
- 首页强调安全状态而不是 popup 风格的紧凑列表

## 视觉风格

Harmony 端整体风格关键词：

- 轻
- 安全
- 克制
- 手机原生
- 微微软系清爽层次
- 带一点鸿蒙沉浸式光感

具体表现为：

- 大标题与顶部留白
- 卡片层次柔和
- 浮岛具备轻光感
- 品牌图标保持真实品牌识别
- 不使用高饱和强调色

## 错误处理

第一阶段首页与本地数据层需要明确处理：

- 本地 vault 读取失败
- 本地 vault 写入失败
- 同步未配置
- 同步失败
- 生物识别失败回退密码

这些错误不应直接导致页面崩溃，而应以状态文案或轻提示方式反馈。

## 验证策略

实现阶段至少验证以下几类：

- 首页预览构建通过
- 本地空 vault 能正常进入首页
- 本地已有账号时能正确渲染卡片
- 添加页保存后返回首页
- 编辑页保存后返回首页
- 删除确认后返回首页

## 范围边界

本规格故意不在这一轮内展开的内容：

- 账号级人工冲突处理 UI
- 后台常驻同步
- 完整 WebDAV 冲突交互
- 高级动画系统
- 大规模品牌资源适配

这些内容后续可以单独出规格，不与首页和本地数据层首版混在一起。
